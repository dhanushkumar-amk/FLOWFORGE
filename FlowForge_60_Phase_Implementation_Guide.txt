PHASE 1 – MONOREPO SETUP & TOOLING CONFIGURATION
────────────────────────────────────────────────────────────────────────────────
GOAL:
  Set up the monorepo workspace so both frontend and backend can coexist,
  share types, and be managed from a single root.

TASKS:
  1. Create root folder: flowforge/
  2. Initialize git: git init
  3. Create root package.json with workspaces config:
       {
         "name": "flowforge",
         "private": true,
         "workspaces": ["apps/*", "packages/*"]
       }
  4. Install Turborepo: npm install turbo --save-dev
  5. Create turbo.json with pipeline for build, lint, dev
  6. Create .gitignore (node_modules, .env, .next, dist, build)
  7. Create .nvmrc with Node version (20.x)
  8. Create root README.md (placeholder for now)
  9. Create folder structure:
       flowforge/
       ├── apps/
       │   ├── frontend/
       │   └── backend/
       ├── packages/
       │   ├── shared-types/
       │   └── config/
       └── .github/
           └── workflows/

FILES TO CREATE:
  - /package.json
  - /turbo.json
  - /.gitignore
  - /.nvmrc
  - /README.md
  - /apps/.gitkeep
  - /packages/.gitkeep

OUTCOME:
  A clean monorepo root. Running "turbo dev" from root will eventually
  start both frontend and backend simultaneously.

PHASE 9 – MONGODB SCHEMAS & MODELS
────────────────────────────────────────────────────────────────────────────────
GOAL:
  Define all MongoDB schemas with proper indexing, validation, and
  TypeScript interfaces. This is the foundation of data integrity.

TASKS:
  1. Create apps/backend/src/models/User.model.ts:
       Schema fields:
         - clerkId: String (unique, required, indexed)
         - email: String (unique, required)
         - name: String (required)
         - avatar: String (optional)
         - plan: enum ['free', 'pro'] (default: 'free')
         - createdAt, updatedAt (timestamps: true)
       Indexes: clerkId (unique), email (unique)

  2. Create apps/backend/src/models/Workspace.model.ts:
       Schema fields:
         - name: String (required)
         - slug: String (unique, required)
         - ownerId: String (required, ref: User)
         - members: [{ userId, role, joinedAt }]
         - plan: enum ['free', 'pro']
         - settings: { maxWorkflows, maxMembers, ... }
         - createdAt, updatedAt
       Indexes: slug (unique), ownerId, members.userId

  3. Create apps/backend/src/models/Workflow.model.ts:
       Schema fields:
         - workspaceId: ObjectId (ref: Workspace, required)
         - name: String (required)
         - description: String
         - dagJson: {
             nodes: [{ id, type, position, data }],
             edges: [{ id, source, target, data }]
           }
         - status: enum ['draft','active','archived']
         - version: Number (default: 1)
         - tags: [String]
         - createdBy: String (clerkId)
         - updatedBy: String
         - isTemplate: Boolean (default: false)
         - createdAt, updatedAt
       Indexes: workspaceId, status, createdBy, tags

  4. Create apps/backend/src/models/Task.model.ts:
       Schema fields:
         - workflowId: ObjectId (ref: Workflow, required)
         - nodeId: String (React Flow node id, required)
         - name: String (required)
         - type: enum ['manual','automated','api','condition','delay','ai']
         - config: Mixed (task-specific config)
         - dependencies: [String] (nodeIds this task depends on)
         - retryPolicy: { maxRetries, delay, backoff }
         - timeout: Number (ms)
         - status: enum ['pending','running','completed','failed','skipped']

  5. Create apps/backend/src/models/Execution.model.ts:
       Schema fields:
         - workflowId: ObjectId (ref: Workflow, required)
         - workspaceId: ObjectId (required)
         - triggeredBy: String (clerkId or 'cron' or 'webhook')
         - status: enum ['pending','running','completed','failed','cancelled']
         - startedAt: Date
         - completedAt: Date
         - duration: Number (ms)
         - stepResults: [{
             nodeId, taskName, status, startedAt,
             completedAt, output, error, retryCount

         - logs: [{ timestamp, level, message, nodeId }]
         - inngestRunId: String
       Indexes: workflowId, workspaceId, status, triggeredBy, startedAt

  6. Create apps/backend/src/models/AuditLog.model.ts:

         - workspaceId: ObjectId (required)

         - action: String (required) e.g. 'workflow.created'
         - resource: String e.g. 'workflow'
         - resourceId: String
         - metadata: Mixed
         - ipAddress: String
         - userAgent: String
         - timestamp: Date (default: now)

PHASE 10 – REPOSITORY PATTERN IMPLEMENTATION
────────────────────────────────────────────────────────────────────────────────
GOAL:
  Implement the Repository Pattern to abstract all database operations.
  Controllers never touch models directly — they go through repositories.

TASKS:
  1. Create apps/backend/src/repositories/base.repository.ts:
       - Generic BaseRepository<T> class
       - Methods:
           findById(id): Promise<T | null>
           findOne(filter): Promise<T | null>
           findMany(filter, options): Promise<T[]>
           create(data): Promise<T>
           updateById(id, data): Promise<T | null>
           deleteById(id): Promise<boolean>
           count(filter): Promise<number>
           paginate(filter, page, limit): Promise<{data, total, page, pages}>

  2. Create apps/backend/src/repositories/user.repository.ts:
       - Extends BaseRepository<IUser>
       - Additional methods:
           findByClerkId(clerkId): Promise<IUser | null>
           findByEmail(email): Promise<IUser | null>
           upsertByClerkId(clerkId, data): Promise<IUser>

  3. Create apps/backend/src/repositories/workspace.repository.ts:
       - Extends BaseRepository<IWorkspace>
       - Additional methods:
           findBySlug(slug): Promise<IWorkspace | null>
           findByMemberId(userId): Promise<IWorkspace[]>
           addMember(workspaceId, userId, role): Promise<IWorkspace>
           removeMember(workspaceId, userId): Promise<IWorkspace>
           updateMemberRole(workspaceId, userId, role): Promise<IWorkspace>

  4. Create apps/backend/src/repositories/workflow.repository.ts:
       - Extends BaseRepository<IWorkflow>
       - Additional methods:
           findByWorkspace(workspaceId, filter): Promise<IWorkflow[]>
           findTemplates(): Promise<IWorkflow[]>
           incrementVersion(id): Promise<IWorkflow>
           searchByTags(tags): Promise<IWorkflow[]>

  5. Create apps/backend/src/repositories/execution.repository.ts:
       - Extends BaseRepository<IExecution>
       - Additional methods:
           findByWorkflow(workflowId, filter): Promise<IExecution[]>
           getExecutionStats(workspaceId): Promise<stats>
           appendLog(id, log): Promise<void>
           updateStepResult(id, nodeId, result): Promise<void>

  6. Create apps/backend/src/repositories/auditLog.repository.ts:
       - Extends BaseRepository<IAuditLog>
       - Additional methods:
           logAction(data): Promise<void>
           findByWorkspace(workspaceId, filter): Promise<IAuditLog[]>

FILES TO CREATE:
  - apps/backend/src/repositories/base.repository.ts
  - apps/backend/src/repositories/user.repository.ts
  - apps/backend/src/repositories/workspace.repository.ts
  - apps/backend/src/repositories/workflow.repository.ts
  - apps/backend/src/repositories/execution.repository.ts
  - apps/backend/src/repositories/auditLog.repository.ts
  - apps/backend/src/repositories/index.ts

OUTCOME:
  All DB operations are abstracted. Switching from MongoDB to Postgres

  This prevents bad data from entering the database.

TASKS:
  1. Create apps/backend/src/validation/schemas/workspace.schema.ts:
       - createWorkspaceSchema: { name (3-50 chars), description? }
       - updateWorkspaceSchema: partial of create
       - inviteMemberSchema: { email, role }

  2. Create apps/backend/src/validation/schemas/workflow.schema.ts:
       - createWorkflowSchema: { name, description?, tags? }
       - updateWorkflowSchema: partial
       - dagJsonSchema: validates nodes and edges structure

  3. Create apps/backend/src/validation/schemas/execution.schema.ts:
       - triggerExecutionSchema: { workflowId, input? }

  4. Create apps/backend/src/middleware/validate.ts:
       - Factory function: validate(schema)
       - Returns middleware that runs zod.parse() on req.body
       - On failure: returns 422 with formatted error array
       - On success: calls next()

  5. Apply validate middleware to all existing routes:
       - POST /api/workspaces → validate(createWorkspaceSchema)
       - PATCH /api/workspaces/:id → validate(updateWorkspaceSchema)
       - POST /api/workspaces/:id/members → validate(inviteMemberSchema)

  6. Create apps/backend/src/utils/formatZodErrors.ts:
       - Converts Zod error format to user-friendly messages

PHASE 14 – TOPOLOGICAL SORT IMPLEMENTATION (KAHN'S ALGORITHM)
────────────────────────────────────────────────────────────────────────────────
GOAL:
  Implement Kahn's algorithm for topological sorting of the DAG.
  This determines the correct order to execute workflow tasks.

TASKS:
  1. Create apps/backend/src/utils/dsa/topologicalSort.ts:

       ALGORITHM (Kahn's):
         Input: nodes (array of nodeIds), edges (array of {source, target})

         Step 1: Build adjacency list
           - For each edge (u → v): adjList[u].push(v)

         Step 2: Calculate in-degree for each node
           - inDegree[v]++ for each edge (u → v)

         Step 3: Initialize queue with all nodes having inDegree = 0

         Step 4: Process queue
           - Dequeue node u
           - Add u to result
           - For each neighbor v of u:
               inDegree[v]--
               if inDegree[v] === 0: enqueue v

         Step 5: If result.length !== nodes.length → cycle detected

         Return: ordered array of nodeIds or throw CycleError

       Complexity: O(V + E) time, O(V + E) space

  2. Create apps/backend/src/utils/dsa/cycleDetection.ts:

       ALGORITHM (DFS-based):
         Track: visited set, recursionStack set

         For each unvisited node:
           dfs(node):
             Mark as visited, add to recursionStack
             For each neighbor:
               If not visited: recurse
               If in recursionStack: CYCLE DETECTED → return path
             Remove from recursionStack

         Return: { hasCycle: boolean, cyclePath?: string[] }

       Complexity: O(V + E) time, O(V) space

  3. Create apps/backend/src/utils/dsa/dagValidator.ts:
       - validateDAG(nodes, edges): runs both cycle detection + topo sort
       - Returns: { valid, executionOrder, errors }

  4. Write comprehensive unit tests:
       - Linear DAG: A → B → C → D
       - Branching DAG: A → B, A → C, B → D, C → D
       - Parallel DAG: A → C, B → C
       - Cycle: A → B → C → A (should throw)
       - Self-loop: A → A (should throw)
       - Disconnected graph (should handle)

PHASE 16 – TOKEN BUCKET RATE LIMITER IMPLEMENTATION

TASKS:
  1. Create apps/backend/src/utils/dsa/tokenBucket.ts:

       CLASS: TokenBucket
         Properties:
           capacity: number (max tokens)
           tokens: number (current tokens)
           refillRate: number (tokens per second)
           lastRefill: number (timestamp ms)

         Methods:
           consume(tokens = 1): boolean
             - Refill tokens since lastRefill:
                 elapsed = now - lastRefill
                 tokensToAdd = elapsed * refillRate / 1000
                 tokens = min(capacity, tokens + tokensToAdd)
                 lastRefill = now
             - If tokens >= requested: subtract, return true
             - Else: return false (rate limited)

           getStatus(): { tokens, capacity, refillRate, nextRefillMs }

       CLASS: CircuitBreaker
         Properties:

           failureCount: number

           - Return lock object with extend() and release()

         release(lock): Promise<void>

           - Extends TTL if still holding lock

         - Cron trigger: lock workflowId to prevent double-trigger
         - File upload processing: lock file during processing
